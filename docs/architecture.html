Architecture
Color System
Layer 1 — Primitive Palette
Tailwind CSS Built-in Colors
gray · violet · red · green · amber · blue — each with shades 50–950
e.g. violet[600] = #7c3aed — just a raw hex, no meaning yet
↓
Layer 2 — Semantic Tokens · src/theme/colors.js
colorTokens
Named roles pointing to palette shades. Only file changed when rebranding for a new app.
e.g. 'primary-high': colors.violet[600] — now the hex has a role name. Rebrand violet → blue? Change only here.
↓
Layer 3 — Tailwind Registration · tailwind.config.js
theme.extend.colors: { ...colorTokens }
Spreads tokens into Tailwind — generates bg-primary-high, text-neutral-subtle, border-error etc.
e.g. spreading primary-high generates the bg-primary-high utility class you can use anywhere
↓
Layer 4 — Component Usage
className props + colorTokens import
Tailwind classes for backgrounds, text, borders. Direct colorTokens values for icon color props and placeholders.
e.g. Button uses "bg-primary-high" — never touches #7c3aed directly. All components update when Layer 2 changes.

Architecture
Typography System
Layer 1 — Font Assets
@expo-google-fonts
dm-sans · space-grotesk — loaded via useFonts() in App.tsx at runtime before any component renders
e.g. SpaceGrotesk_700Bold — raw font variable name, no meaning yet
↓
Layer 2 — Token Mapping · src/theme/typography.js
typographyTokens
fontFamily maps role names (heading, base) to font variable names. fontSize maps variant names to px values + line heights.
e.g. 'heading-bold': [SpaceGrotesk_700Bold] — font variable gets a role name. Swap fonts for a new app? Change only here.
↓
Layer 3 — Tailwind Registration · tailwind.config.js
theme.extend.fontFamily + fontSize
Generates: font-heading-bold, font-base-semibold, text-display, text-label etc.
e.g. spreading heading-bold generates the font-heading-bold utility class you can use anywhere
↓
Layer 4 — Text Component · variant + weight props
weightMap[variant][weight] + variantSizes[variant]
variant picks size class + font family. weight picks the specific weight within that family. Combined in className via cn().
e.g. variant="display" weight="bold" → "text-display font-heading-bold" — never references the font file directly

Architecture
Spacing System
Layer 1 — Token Definition · src/theme/spacing.js
spacingTokens
1:1 label-to-px mapping — token 16 = 16px. No mental math.
e.g. 16: '16px' — the number is exactly what you get in pixels
↓
Layer 2 — Tailwind Registration · tailwind.config.js
theme.extend.spacing: spacingTokens
Generates: p-16 = 16px, m-8 = 8px, gap-24 = 24px etc.
e.g. spreading token 16 generates p-16, px-16, gap-16, w-16 — all at once
↓
Layer 3 — Component Usage
className props — p-*, m-*, gap-*, w-*, h-* etc.
Any Tailwind utility that accepts spacing uses these tokens: padding, margin, gap, width, height, inset.
e.g. Button uses px-16 py-12 — reads as exactly 16px horizontal, 12px vertical padding

Border Radius System

radiusTokens

↓

className props — rounded-* utilities
Used on containers, buttons, inputs, cards. Any rounded-* utility accepts these tokens.
e.g. Button uses rounded-12, Toggle track uses rounded-1000 — same token scale across all components

Architecture
Shadow System
Layer 1 — Token Definition · src/theme/shadows.js
shadowTokens + shadowStyles
3 named levels — subtle · medium · strong. Each exports a CSS string (for Tailwind) and a RN style object (for direct spread).
e.g. medium: { shadowOffset: {0,4}, opacity: 0.10, radius: 12, elevation: 6 }
↓
Layer 2 — Tailwind Registration · tailwind.config.js
theme.extend.boxShadow: shadowTokens
Generates: shadow-subtle, shadow-medium, shadow-strong — NativeWind compiles these to iOS shadow props + Android elevation.
e.g. spreading medium generates the shadow-medium utility class
↓
Layer 3 — Component Usage
className="shadow-*" or ...shadowStyles.* spread
Regular Views use the className. Animated.Views (Dialog, FilterSheet, Dropdown) spread the style object directly since Animated doesn't support NativeWind className.
e.g. BottomNav uses ...shadowStyles.medium · Dialog uses ...shadowStyles.strong

Platform
iOS
Uses shadowColor + shadowOffset + shadowOpacity + shadowRadius. All 4 props needed for shadow to appear.
Android
Uses elevation only — a single integer that controls drop shadow depth. The other shadow props are ignored.
Special case — FilterSheet
Bottom sheet needs an upward shadow (height: -4). Uses ...shadowStyles.strong then overrides shadowOffset for the direction.
